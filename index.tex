\section{Introduction}\label{introduction}

Converting standalone markdown documentation into a professional blog
post presents a common challenge: a well-written technical reference
with working code examples and thorough coverage sits as a flat file on
disk, invisible to anyone who might benefit from it.

The gap between a markdown file and a published blog post turns out to
be narrower than expected, but only if a systematic workflow is
followed. The ZZCOLLAB framework provides the scaffolding: symlinks
bridge Quarto's expectations with the rrtools research compendium
layout, and a reproducible Docker environment ensures that readers can
rebuild the post from source.

This post documents that conversion workflow from start to finish,
including the decisions required, the directory structure to create, and
the verification steps that prevent broken links and missing images from
reaching the audience.

\subsection{Motivations}\label{motivations}

The following considerations motivated this workflow:

\begin{itemize}
\tightlist
\item
  Hundred-line markdown documents sitting in project directories remain
  invisible to anyone outside the project.
\item
  Search engines cannot index a markdown file on a local disk; a
  published blog post reaches a broader audience.
\item
  A repeatable process allows converting future documentation in minutes
  rather than hours.
\item
  The ZZCOLLAB dual-symlink system already solves the
  Quarto-versus-rrtools path conflict; documenting its use for
  documentation conversion specifically fills a gap.
\item
  A pre-publication checklist prevents publishing posts with broken
  image paths and missing YAML fields.
\end{itemize}

\subsection{Objectives}\label{objectives}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Walk through the complete workflow for converting a standalone
  markdown file into a ZZCOLLAB-compatible Quarto blog post.
\item

\item

\section{What is the ZZCOLLAB Blog
Workflow?}\label{what-is-the-zzcollab-blog-workflow}

The ZZCOLLAB blog workflow is a systematic process for converting
standalone markdown documentation into a Quarto blog post that lives
inside a reproducible research compendium. At its core, it solves a path
conflict: Quarto expects blog posts at \texttt{posts/*/index.qmd}, while
rrtools places authored content in \texttt{analysis/paper/}. The
dual-symlink system bridges both conventions so that a single source
file satisfies both tools.

Think of it as a translation layer. Your markdown content goes into the
rrtools-standard location. Symlinks at the project root make Quarto
believe the file is where it expects. Image paths like
\texttt{media/images/hero.jpg} resolve correctly from both locations
because symlinks exist in both directions.

The workflow itself takes roughly 40 minutes from start to finish: five
minutes for assessment, three for initialisation, one for symlink setup,
ten for content conversion, five for media assets, three for metadata,
five for testing, and two for committing.

\section{Getting Started}\label{getting-started}

\subsection{Step 1: Assess Your Source
Document}\label{step-1-assess-your-source-document}

Before touching any tools, spend five minutes evaluating the markdown
file you plan to convert. Ask yourself:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  How long is the document? A 200-line reference differs from a 600-line
  tutorial.
\item
  Does it contain code examples? If so, in which languages?
\item
  Does it need images? Hero images, diagrams, screenshots?
\item
  Is it a tutorial, reference guide, or how-to?
\item
  Who is the target audience?
\end{enumerate}

\begin{verbatim}
Example assessment:
- Length: ~600 lines (substantial)
- Code: Yes (bash scripts, R snippets)
- Images: Yes (hero + concept diagram)
- Type: How-to guide with reference sections
- Audience: R developers using ZZCOLLAB
- Estimate: 10 minutes to convert content
\end{verbatim}

\section{Integration with the Parent
Blog}\label{integration-with-the-parent-blog}

The parent Quarto blog configuration (\texttt{\_quarto.yml}) already
renders posts matching the pattern \texttt{posts/*/index.qmd}. Because
the root-level symlink maps \texttt{index.qmd} to
\texttt{analysis/paper/index.qmd}, the blog discovers your post
automatically.

No changes to the parent blog configuration are required. When you
render the entire site with \texttt{quarto\ render}, the new post
appears in the listing.

\subsection{Things to Watch Out For}\label{things-to-watch-out-for}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \textbf{Symlinks break after cloning.} If someone clones your
  repository and the symlinks do not recreate correctly, they must run
  \texttt{setup\_symlinks.sh} again or recreate them manually.
\item
  \textbf{Tabs in YAML cause silent failures.} Quarto YAML requires
  spaces for indentation. A single tab character produces a parse error
  that does not always point to the offending line.
\item
  \textbf{Absolute image paths break portability.} Always use relative
  paths (\texttt{media/images/hero.jpg}), never absolute paths. Absolute
  paths work on your machine and nowhere else.
\item
  \textbf{The \texttt{draft} field defaults to \texttt{false} when
  omitted.} If you forget to set \texttt{draft:\ false}, your unfinished
  post appears on the public site after the next render.
\item
  \textbf{Large images slow rendering and page load.} Keep images under
  500 KB each. Use JPEG for photographs and PNG for diagrams with sharp
  edges.
\end{enumerate}

\item
  Blog posts are more discoverable than standalone markdown files; the
  effort of conversion pays off in audience reach.
\item
  Structured YAML metadata enables Quarto's listing features (search,
  category filtering, date sorting) without custom JavaScript.
\item
  The rrtools research compendium layout provides a natural separation
  between authored content, analysis code, and media assets.
\end{itemize}

\subsubsection{Technical Skills}\label{technical-skills}

\begin{itemize}
\tightlist
\item
  Setting up symlinks with \texttt{ln\ -s} and verifying them with
  \texttt{ls\ -la} became second nature after the first conversion.
\item
  Writing Quarto callouts (\texttt{:::\ \{.callout-note\}\ ...\ :::})
  replaced blockquotes for emphasis and improved visual hierarchy.
\item
  The \texttt{.img-fluid} class in Bootstrap ensures images scale
  correctly on mobile without any custom CSS.
\item
  Using \texttt{quarto\ render} for local preview before committing
  caught broken paths that would otherwise reach the public site.
\end{itemize}

\subsubsection{Gotchas and Pitfalls}\label{gotchas-and-pitfalls}

\begin{itemize}
\tightlist
\item
  Forgetting to document image sources in
  \texttt{analysis/media/images/README.md} requires reconstructing
  attribution after the fact.
\item
  A missing \texttt{.img-fluid} class causes hero images to overflow
  their container on mobile devices.
\item
  Committing a post with \texttt{draft:\ false} by accident results in
  the post appearing live on the site before the metadata is corrected.
\item
  Quarto's error messages for broken image paths are sometimes vague;
  using \texttt{grep} for all image references before rendering saves
  debugging time.
\end{itemize}

\subsection{Limitations}\label{limitations}

\begin{itemize}
\tightlist
\item
  This workflow assumes a Unix-like environment (macOS or Linux) for
  symlink support. Windows users may need WSL or alternative approaches.
\item
  The 40-minute estimate applies to straightforward documentation with
  minimal code and few images. Complex tutorials with interactive
  elements take longer.
\item
  The dual-symlink system adds conceptual complexity that may confuse
  contributors unfamiliar with the convention.
\item
  ZZCOLLAB must be installed and functional before starting.
  Troubleshooting ZZCOLLAB itself is outside the scope of this workflow.
\item
  The workflow does not cover adding R analysis pipelines to posts that
  are purely narrative; extending a documentation post with data
  analysis requires additional steps.
\item
  Local rendering requires Quarto installed on the author's machine.
  Docker-based rendering is possible but adds setup time.
\end{itemize}

\end{enumerate}

\section{Wrapping Up}\label{wrapping-up}

Converting standalone markdown documentation into a ZZCOLLAB blog post
is a systematic, repeatable process that takes roughly 40 minutes once
you understand the workflow. The dual-symlink system bridges the gap
between Quarto's directory expectations and the rrtools research
compendium layout, and proper YAML metadata ensures your content is
discoverable through search and category filtering.

The key insight from developing this workflow is that the barrier to
publishing is not the conversion itself but the lack of a structured
process. Once the steps are documented and the checklist is in place,
each subsequent conversion is faster and less error-prone.

For those with documentation sitting in project directories that could
benefit a broader audience, this workflow merits consideration. The
initial investment in understanding the symlink system pays dividends
every time another document is converted.

Main takeaways:

\begin{itemize}
\tightlist
\item
  The dual-symlink system is the architectural foundation; understand it
  first.
\item
  YAML metadata drives discoverability; invest time in titles,
  descriptions, and categories.
\item
  A pre-publication checklist prevents the most common failures.
\item
  The entire process takes about 40 minutes from assessment to committed
  post.
\end{itemize}
